# Agent 资源管理平台产品功能规格

> 状态：待产品确认  
> 版本：v0.1  
> 面向对象：团队内部 Agent 构建者、使用者与平台管理员  
> 本文只定义产品能力与用户体验，不约束具体技术实现。

## 1. 产品定位

Agent 资源管理平台是团队内部统一的 Agent 控制平面，用于管理 Agent 的运行环境、配置资源、调试过程和发布生命周期。

平台解决五个核心问题：

1. 团队有哪些可以运行 Agent 的物理环境，它们是否可用。
2. 每台环境安装、运行了哪些 Agent Runtime，例如 OpenCode。
3. Runtime 中已经存在的 Agent、Skill、MCP 配置是什么，是否与平台一致。
4. 如何快速创建、调试、评估和发布一个 Agent。
5. Agent 运行时做了什么、调用了什么工具、结果是否合格、出现问题后能否追踪和回滚。

平台首页首先是一个可直接工作的多 Agent 对话工作台，而不是资源统计仪表盘。资源管理和 Agent Studio 是支撑对话与运行的管理能力。

## 2. 已确认边界

- 产品形态：团队内部平台，不按对外多租户 SaaS 设计。
- 节点范围：本机与远程 Linux 服务器。
- 远程接入：第一阶段采用 SSH。
- Agent 配置真源：平台数据库中的 `Agent`。
- OpenCode 定位：一种可发现、可加载、可发布的 Agent Runtime Provider。
- 同步原则：
  - 发现时从 OpenCode 导入平台。
  - 平台修改形成新版本，不立即覆盖服务器文件。
  - 发布时由平台将已确认版本写入目标环境。
  - 双方均有修改时必须人工处理冲突。
- 展示 Agent 推理过程时，只展示结构化步骤、思考摘要、工具调用和评估结果，不展示模型私有思维链原文。

## 3. 产品目标与成功标准

### 3.1 产品目标

- 新用户可以在 10 分钟内发现本机 OpenCode，并选择一个已有 Agent 开始对话。
- 管理员可以注册远程 Linux 节点，看到连接状态、Runtime 和配置资源。
- Agent 构建者不直接编辑服务器文件，也能完成创建、调试、评估、发布、下线和回滚。
- 用户能看懂 Agent 当前在做什么，并可审批高风险工具调用。
- 每次配置变更、发布和远程操作都有责任人、时间、目标和结果记录。

### 3.2 MVP 成功标准

- 完成“节点注册 → OpenCode 发现 → 资源导入 → Agent 调试 → Eval → 发布 → 对话 → 下线/回滚”的闭环。
- 同一 Agent 在全平台使用同一个 ID，不再区分 Looper Agent 和 Runtime Agent。
- 所有远程操作和发布操作都可审计。
- 任何凭据都不在页面、日志和普通 API 响应中显示明文。
- 发布版本可以明确回答：谁在何时将哪个版本发布到了哪些节点。

## 4. 用户角色

### 4.1 平台管理员

负责节点、凭据、成员权限、Runtime Provider 和全局策略。

主要操作：

- 注册、维护、停用和删除计算节点。
- 管理 SSH 凭据引用和主机信任关系。
- 执行全节点发现、处理异常节点。
- 查看全部发布、运行和审计记录。
- 配置组织级工具审批、安全和发布策略。

### 4.2 Agent 构建者

负责创建、配置、调试、评估和发布 Agent。

主要操作：

- 从模板、OpenCode 导入或空白创建 Agent。
- 绑定 Skill、MCP、子 Agent 和数据资源。
- 配置 Loop、SOP、Hook、Eval 和 Guardrail。
- 在沙盒中调试，提交发布或下线。
- 查看自己负责 Agent 的运行表现。

### 4.3 Agent 使用者

在对话工作台选择已发布 Agent 完成任务。

主要操作：

- 创建和管理会话。
- 选择有权限的 Agent。
- 查看执行步骤、工具调用和结果。
- 审批被策略标记为需要人工确认的工具调用。
- 对结果进行反馈。

### 4.4 审计者

只读查看配置版本、发布、远程操作、工具审批和运行记录，不能修改资源或执行任务。

## 5. 统一术语

### 5.1 计算节点 Compute Node

可以承载 Agent Runtime 的物理运行环境。第一阶段包括：

- Local：平台所在本机。
- Remote Linux：通过 SSH 管理的远程 Linux 服务器。

节点不是 Agent，也不是容器。它负责提供运行位置、环境信息、健康状态和远程操作边界。

### 5.2 Agent Runtime

安装在计算节点上、可以加载和执行 Agent 的运行时。第一阶段重点支持 OpenCode，后续可扩展 OpenClaw、Harness 和自定义 Runtime。

现有代码中的 `AgentContainer` 在用户界面统一显示为“Agent Runtime”，避免将所有 Runtime 误解为 Docker 容器。

### 5.3 Agent

可配置、可版本化、可调试和可发布的智能体定义。Agent 包含目标、提示词、模型、工具资源、协作方式、执行策略、安全规则和评估规则。

### 5.4 Skill

提供知识、规范或任务方法的能力模块，第一阶段主要对应 `SKILL.md`。

### 5.5 MCP

Agent 可连接和调用的外部工具服务，包括 stdio、HTTP、SSE 等形态。MCP 只是工具连接，不等于 Skill。

### 5.6 Agent Version

Agent 配置的不可变快照。每次保存可产生草稿修订；发布时固定为明确版本。

### 5.7 Deployment

将某个 Agent Version 加载到一个或多个目标 Runtime 的记录。

### 5.8 Run

一次 Agent 实际执行。短对话和长任务都产生 Run，只是交互方式和持续时间不同。

### 5.9 Session

用户与某个 Agent 的连续对话上下文，可包含多个 Run。

## 7. 核心产品能力

## 7.1 计算节点管理

### 注册本机

- 自动读取主机名、操作系统、架构和平台版本。
- 检测节点是否已经注册，重复操作只更新状态，不创建重复节点。
- 注册后立即进入 Runtime 发现预览。

### 注册远程 Linux

采用五步向导：

1. 基本信息：名称、主机地址、端口、标签和所属环境。
2. 身份认证：选择已有凭据或创建新的凭据引用。
3. 主机信任：展示 SSH 主机指纹，由管理员明确确认。
4. 权限检查：检查连接、基础命令、配置目录读取和发布目录写入权限。
5. 发现预览：展示 Runtime 和资源，不自动导入。

### 节点详情

展示：

- 基本信息与标签。
- 在线、离线、维护中、凭据失效等状态。
- 最近心跳和最近扫描。
- 已发现 Runtime。
- 当前部署的 Agent。
- 近期远程操作和失败原因。

可执行：

- 测试连接。
- 重新扫描。
- 更换凭据引用。
- 进入或退出维护模式。
- 停用。
- 删除。

删除前必须处理该节点上的有效部署和运行中任务。

## 7.2 Runtime 与资源自动发现

### 发现层级

一次扫描依次产生：

1. 节点环境信息。
2. 已安装和运行中的 Agent Runtime。
3. Runtime 版本、路径、进程和健康状态。
4. Agent 配置。
5. Skill 配置。
6. MCP 配置。

### 发现结果处理

每项发现结果必须标识为：

- 新发现：平台不存在，可以导入。
- 已关联：与平台内容一致。
- 有差异：平台和服务器内容不同。
- 缺失：平台记录存在，但服务器文件已不存在。
- 不支持：能够识别，但当前 Provider 无法管理。
- 错误：读取或解析失败。

用户可以：

- 查看差异。
- 导入为新资源。
- 关联到已有资源。
- 忽略。
- 标记为由服务器外部管理。
- 对冲突选择“保留平台版本”“导入为新版本”或“暂不处理”。

不提供无提示的“最新时间覆盖”。

### OpenCode Provider

第一阶段支持发现：

- OpenCode 是否安装。
- OpenCode CLI 路径和版本。
- OpenCode 相关进程。
- Agent 配置文件。
- `SKILL.md`。
- MCP 配置。

平台中的 OpenCode 资源保留来源节点、来源路径和最后发现时间，但导入后由平台版本体系管理。

## 7.3 Agent 生命周期管理

### Agent 状态

```text
草稿 draft
  → 已加载 loaded
  → 调试中 testing
  → 待发布 ready
  → 已发布 published
  → 已下线 offline
  → 已归档 archived
```

补充规则：

- 调试失败不自动改变为下线，只标记当前版本未通过门禁。
- 已发布 Agent 可以继续创建新草稿，不影响线上版本。
- 下线针对 Deployment，不删除 Agent 和历史版本。
- 归档后不可新建会话，但历史记录保留。

### 创建方式

- 从空白创建。
- 从团队模板创建。
- 从已发现 OpenCode Agent 导入。
- 复制现有 Agent。
- 从历史版本分支创建。

### 加载

“加载”表示将某个 Agent 草稿或版本放入指定 Runtime 的调试环境，使其可以接受测试请求，但不对普通用户开放。

加载前检查：

- 目标 Runtime 在线。
- 模型和凭据可用。
- Skill、MCP 和子 Agent 依赖完整。
- 配置格式符合 Provider 要求。
- 高风险工具策略已经设置。

### 调试

- 在 Agent Studio 内直接发起流式对话。
- 可以临时覆盖输入上下文，但不修改正式版本。
- 展示步骤、工具、子 Agent、Eval 和错误。
- 可以将某次成功 Run 保存为回归样例。
- 可以将失败 Run 复制为调试用例。

### 发布

发布流程：

1. 固化 Agent Version。
2. 查看相对当前线上版本的差异。
3. 选择目标节点和 Runtime。
4. 执行依赖检查。
5. 执行必选 Eval 套件。
6. 根据权限直接发布或提交审批。
7. 展示逐目标发布进度。
8. 生成 Deployment 和审计记录。

### 下线

- 可以下线单个 Deployment，也可以下线 Agent 的全部 Deployment。
- 有运行中任务时，默认禁止立即下线。
- 管理员可以选择“停止接收新任务，等待现有任务完成”。
- 强制下线需要二次确认和原因。
- 下线后不删除服务器历史备份和平台版本记录。

### 回滚

- 选择同一 Agent 的历史已发布版本。
- 发布前展示目标、依赖和配置差异。
- 回滚本质是一次新的 Deployment 操作，不修改历史记录。

## 7.4 Agent 对话工作台

### 会话与 Agent 选择

- 一个会话固定关联一个 Agent 和一个已发布版本。
- 切换 Agent 时默认新建会话，避免不同 Agent 上下文混用。
- 构建者可在调试会话中选择草稿版本。
- Agent Picker 只展示用户有权限使用且状态可用的 Agent。

### 消息体验

支持：

- 用户消息。
- Agent 最终回复。
- 附件或结构化上下文入口。
- 流式输出。
- 停止生成。
- 重新生成。
- 复制结果。
- 对结果进行有用/无用反馈。

### 执行轨迹

执行轨迹按时间顺序显示：

- 已接收任务。
- 规划或步骤摘要。
- 子 Agent 委派和返回。
- 工具调用名称、参数摘要和结果摘要。
- 等待审批。
- Eval 分数、结论和反馈。
- 重试与修订。
- 完成、取消或失败。

默认只显示摘要。构建者和管理员可以展开结构化输入输出；敏感字段始终脱敏。

### 工具审批

工具策略分为：

- 自动允许。
- 每次询问。
- 按条件询问。
- 禁止。

以下操作默认至少需要“每次询问”：

- 写文件。
- 执行 Shell。
- 修改数据库。
- 对外发送消息。
- 创建、修改或删除外部资源。

审批卡展示工具、目的、目标资源、参数摘要和风险等级。审批可以允许一次、拒绝或在有权限时调整该会话策略。

## 7.5 Agent Studio 与设计方法论

Agent Studio 不是 JSON 编辑器，而是一套显式的 Agent 设计流程。

### 第一步：任务目标

需要定义：

- Agent 要解决什么问题。
- 输入和输出是什么。
- 什么结果算成功。
- 哪些事情明确不做。

完成结果：目标卡和成功标准。

### 第二步：角色与指令

需要定义：

- Agent 角色。
- 工作原则。
- 输出格式。
- 领域术语。
- 冲突时的优先级。

完成结果：系统提示词草稿。专家模式可以查看和编辑完整文本。

### 第三步：模型与上下文

需要定义：

- 使用的模型。
- 采样参数。
- 会话记忆窗口。
- 可以读取的数据和知识范围。

产品需提示成本、速度和能力差异，但第一阶段不做自动模型路由。

### 第四步：能力资源

绑定：

- Skill。
- MCP。
- 平台内置工具。
- 数据源或知识库。
- 凭据引用。

每个资源显示来源、权限、可用节点和风险等级。

### 第五步：协作拓扑

MVP 支持五种模式：

1. Single：单 Agent 完成任务。
2. Sequential：多个子 Agent 顺序处理。
3. Router：主 Agent 根据任务选择子 Agent。
4. Parallel：多个子 Agent 并行处理，再合并结果。
5. Evaluator-Optimizer：执行者产出，评估者给出反馈，执行者修订。

每种模式提供：

- 适用场景。
- 最小示例。
- 成本和延迟提示。
- 默认的失败处理策略。

第一阶段采用步骤式配置和拓扑预览，不提供任意拖拽 DAG。

### 第六步：Loop 与 SOP

Loop 定义单个 Agent 如何反复工作；SOP 定义多个步骤或 Agent 如何协作。

支持的 Loop：

- Single Shot。
- ReAct。
- Plan and Execute。
- Reflect。
- Evaluator-Optimizer。

SOP 步骤类型：

- Agent。
- Tool。
- Human Approval。
- Condition。
- Parallel。
- Merge。
- Notification。

MVP 只允许无环流程；复杂循环通过 Loop 策略表达。

### 第七步：Hook

Hook 是运行生命周期中的受控扩展点，不是任意 Shell 脚本。

MVP Hook 点：

- Run Before / After。
- Step Before / After。
- Tool Before / After。
- Error。
- Eval After。
- Publish Before / After。

Hook 动作：

- 写审计事件。
- 注入上下文。
- 校验输入或输出。
- 脱敏。
- 通知。
- 触发人工审批。
- 中断运行或发布。

### 第八步：Eval

Eval 用来回答“这个 Agent 是否达到了目标”，而不仅是观察它是否报错。

Eval 类型：

- 规则：字段、格式、阈值、关键词和确定性断言。
- LLM Judge：按评分标准输出分数、结论和反馈。
- 人工：由指定角色确认。
- 混合：多个 Eval 按全部通过、加权分数或关键项否决组合。

Eval 结论统一为：

- PASS：通过。
- REVISE：需要修订并可进入下一轮。
- FAIL：失败。
- NEEDS_REVIEW：需要人工判断。

Eval 套件可以标记为：

- 调试可选。
- 发布必选。
- 定时回归。

### 第九步：Guardrail

至少配置：

- 最大运行时间。
- 最大迭代数。
- 最大工具调用数。
- 最大并行子 Agent 数。
- 可访问资源范围。
- 高风险工具审批。
- 失败、超时和成本超限的处理方式。

### 第十步：调试与发布

Studio 最后展示配置完整度和发布准备度：

- 配置完整。
- 依赖完整。
- 安全策略完整。
- Eval 已通过。
- 目标 Runtime 可用。

只有满足组织发布策略后才允许发布。

## 7.6 Skill 与 MCP 管理

### Skill

- 从 Runtime 自动发现。
- 查看 Markdown 内容、来源、版本和引用 Agent。
- 导入为平台资源。
- 创建平台版本。
- 绑定或解绑 Agent。
- 发布到目标 Runtime。
- 检查同名冲突。

### MCP

- 从 OpenCode 配置自动发现。
- 展示传输方式、命令或地址、环境变量名称和健康状态。
- 敏感环境变量只显示引用，不显示值。
- 支持连通性测试。
- 绑定或解绑 Agent。
- 发布前进行依赖检查。

MVP 不提供“输入任意 API 后自动生成 MCP Server 代码”，只管理已有 MCP 配置和连接。

## 7.7 运行、监控与审计

### 运行中心

统一展示短对话 Run 和长任务 Run：

- 排队中。
- 运行中。
- 等待审批。
- 已暂停。
- 需复核。
- 已完成。
- 已失败。
- 已取消。

支持按 Agent、用户、节点、状态和时间筛选。

### 运行操作

- 查看实时轨迹。
- 暂停和恢复支持该能力的长任务。
- 取消运行。
- 重试失败步骤。
- 从 checkpoint 恢复。
- 将运行保存为 Eval 样例。

### 审计事件

至少记录：

- 节点和凭据引用的创建、修改、测试和删除。
- 扫描与发现。
- 资源导入和冲突处理。
- Agent 配置与版本变更。
- 加载、发布、下线和回滚。
- 工具调用审批。
- 高风险远程操作。
- 权限和组织策略变更。

审计记录包含操作者、时间、对象、动作、来源、结果和关联 Run/Deployment，不能由普通用户修改。

## 8. 产品状态模型

## 8.1 计算节点状态

- Pending：已创建，尚未完成连接验证。
- Online：连接与基础检查正常。
- Degraded：可以连接，但部分能力异常。
- Offline：无法连接。
- Maintenance：管理员主动进入维护。
- Disabled：已停用，不允许扫描、发布和运行。

状态由系统检测与管理员动作共同决定，不能用手工“在线”覆盖真实连接失败。

## 8.2 Runtime 状态

- Discovered：已发现，尚未纳管。
- Managed：已纳管。
- Running：Runtime 进程可用。
- Stopped：已安装但未运行。
- Error：版本、进程或配置异常。
- Unsupported：已识别但 Provider 不支持管理。

## 8.3 Deployment 状态

- Pending。
- Validating。
- Deploying。
- Active。
- Draining。
- Offline。
- Failed。
- RolledBack。

## 8.4 Run 状态

- Queued。
- Running。
- AwaitingApproval。
- Paused。
- NeedsReview。
- Completed。
- Failed。
- Cancelled。

## 9. 权限产品规则

MVP 采用角色加资源范围：

- 平台管理员：全部权限。
- Agent 构建者：管理授权范围内的 Agent、版本、调试和发布。
- Agent 使用者：使用已授权的已发布 Agent。
- 审计者：只读查看运行、发布和审计。

高风险操作需要单独权限：

- 注册远程节点。
- 创建或修改凭据引用。
- 执行远程扫描。
- 发布到生产标签节点。
- 强制停止 Run。
- 强制下线。
- 批准高风险工具。

权限不足时不应只隐藏按钮，服务端也必须拒绝操作；具体技术实现留到技术方案阶段。

## 10. 核心用户旅程

## 10.1 注册本机并开始对话

1. 管理员点击“注册本机”。
2. 平台展示环境和 OpenCode 发现预览。
3. 管理员确认导入已有 Agent、Skill 和 MCP。
4. Agent 出现在 Agent 中心。
5. 构建者加载并完成一次调试。
6. 发布后，使用者在首页选择该 Agent 开始对话。

## 10.2 注册远程 Linux 节点

1. 管理员填写节点和凭据引用。
2. 确认主机指纹。
3. 平台执行权限检查。
4. 展示 OpenCode 和配置发现结果。
5. 管理员确认导入。
6. 节点进入 Online 或 Degraded。

## 10.3 处理 OpenCode 配置冲突

1. 扫描发现服务器文件与平台版本不同。
2. 资源标记为“有差异”。
3. 用户查看结构化 diff。
4. 选择保留平台版本、导入为新版本或暂不处理。
5. 所有选择写入审计。

## 10.4 从模板创建 Agent

1. 构建者选择场景模板。
2. 按 Agent Studio 方法论完成配置。
3. 系统提示缺失依赖和风险项。
4. 加载到调试 Runtime。
5. 进入对话调试。

## 10.5 对话中审批工具

1. Agent 请求高风险工具。
2. Run 进入 AwaitingApproval。
3. 用户看到工具、目的、参数摘要和风险。
4. 用户允许一次或拒绝。
5. Agent 继续执行或调整方案。

## 10.6 Eval 后发布

1. 构建者提交发布。
2. 平台运行发布必选 Eval。
3. Eval 给出 PASS、REVISE、FAIL 或 NEEDS_REVIEW。
4. 通过后选择目标节点并发布。
5. 发布完成后新会话使用新版本，已有会话保持原版本。

## 10.7 下线

1. 构建者选择 Deployment 下线。
2. 平台检查活跃 Run。
3. 默认进入 Draining，不再接收新任务。
4. 活跃 Run 完成后进入 Offline。
5. 强制下线必须填写原因。

## 10.8 回滚

1. 构建者从发布历史选择稳定版本。
2. 查看差异和依赖检查。
3. 执行回滚发布。
4. 新 Deployment 生效，原 Deployment 保留历史记录。

## 11. MVP 范围

## 11.1 MVP 必须完成

- 默认多 Agent 对话首页。
- 本机与 SSH Linux 节点注册。
- 节点连接检查、状态和扫描。
- OpenCode 安装、进程、Agent、Skill 和 MCP 发现。
- 发现预览、差异、确认导入和人工冲突处理。
- Agent 单一真源、版本和状态。
- Agent 加载、流式调试、发布、下线和回滚。
- Skill/MCP 绑定和依赖检查。
- Single、Sequential、Router、Parallel、Evaluator-Optimizer 协作模式。
- 五种基础 Loop。
- 基础 Hook、Eval 和 Guardrail。
- 工具调用审批。
- Run 轨迹、取消、失败和需复核状态。
- 角色权限与审计。

## 11.2 MVP 后续增强

- OpenClaw、Harness 和自定义 Runtime Provider。
- 灰度比例、自动扩缩和跨节点调度。
- Eval 数据集管理和趋势分析。
- 可视化 DAG 编辑器。
- 定时任务和事件触发 Agent。
- Agent 市场和跨团队共享审批。
- 成本预算和模型自动路由。
- Kubernetes 原生节点和工作负载管理。

## 11.3 明确不做

- 对外多租户计费 SaaS。
- 任意 Shell 命令编排器。
- 展示模型私有思维链。
- 第一阶段支持 Windows 远程节点。
- 自动以“最后更新时间”为依据覆盖 OpenCode 配置。
- 首版自研通用 AI SDK。
- 首版完整实现任意有环 DAG。
- 根据任意 HTTP API 自动生成并托管 MCP Server。

## 12. 产品验收标准

### 节点与发现

- 本机重复注册不会产生重复节点。
- SSH 节点未确认主机指纹前不能扫描或发布。
- 扫描可以明确区分新资源、一致、差异、缺失和失败。
- 导入前不会修改远程 OpenCode 配置。

### Agent 生命周期

- 一个 Agent 在列表、Studio、对话、运行和发布中使用相同身份。
- 发布必须绑定不可变版本。
- 下线不删除 Agent、版本、会话和运行历史。
- 回滚产生新的部署记录。

### 对话与执行

- 用户可以选择已授权 Agent 并收到流式回复。
- 用户可以看到结构化执行轨迹。
- 需要审批的工具在批准前不能执行。
- 用户可以取消运行并看到最终状态。
- 断开页面后可以重新进入查看已持久化轨迹。

### Agent Studio

- 新用户可以从模板完成一个 Single Agent。
- 专家可以配置子 Agent 协作、Hook、Eval 和 Guardrail。
- 发布前可以明确看到未满足的门禁。
- Eval 的评分标准和结论可追溯。

### 安全与审计

- 普通用户不能注册节点、查看凭据或发布未授权 Agent。
- 页面和普通响应不出现 SSH、MCP 或模型密钥明文。
- 节点扫描、发布、下线、回滚和工具审批均有审计记录。

## 13. 原型阶段必须验证的问题

1. 默认首页三栏布局是否适合高频对话，右侧轨迹是否会造成信息过载。
2. “Agent Runtime”是否比“智能体容器”更容易理解。
3. Agent Studio 的十步方法论是否过长，快速模式应合并哪些步骤。
4. 发现冲突的 diff 和处理动作是否足够清晰。
5. 加载、发布、下线、归档四个概念是否容易区分。
6. Loop、SOP、Hook、Eval 的解释和示例是否能让首次用户理解。
7. 工具审批卡是否同时满足安全性与操作效率。
8. 资源管理和对话工作台之间的跳转是否自然。
