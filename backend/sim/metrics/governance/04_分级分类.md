# 指标分级分类

## 一、指标分级（Level）

按业务重要性和使用范围分三级：

### L1 一级指标

- **对象**：全公司核心 KPI，CEO/CFO/CRO 关注
- **数量**：20–30 个
- **举例**：`disburse_amount`（放款金额）、`overdue_rate_m1`、`net_profit`
- **管理**：任何变更需 MGC 审批
- **考核**：与年度/季度目标挂钩

### L2 二级指标

- **对象**：部门核心 KPI，各业务线负责人关注
- **数量**：60–100 个
- **举例**：各产品的通过率、各渠道 CPA、部门费用率
- **管理**：Owner + Steward 审批
- **考核**：部门/团队季度考核

### L3 三级指标

- **对象**：明细分析指标，一线业务和分析师关注
- **数量**：100+ 个
- **举例**：各年龄段/学历/职业客户分布、各分公司业绩
- **管理**：Owner 单批
- **考核**：不直接与考核挂钩，服务分析

## 二、指标分类（Domain）

按业务领域横向切分：

| Domain 代码 | 中文 | 举例指标 |
|---|---|---|
| `user` | 用户域 | DAU、MAU、留存率 |
| `credit` | 信贷域 | 通过率、放款额、支用率 |
| `finance` | 财务域 | 净利润、净息差、ROA |
| `risk` | 风险域 | 逾期率、Vintage、反欺诈拦截率 |
| `marketing` | 营销域 | CPA、ROI、渠道转化率 |
| `operation` | 经营管理域 | 人均产能、客服接通率 |
| `funding` | 资金/合作方域 | 资金成本、分润金额 |
| `compliance` | 合规域 | 投诉率、监管报送及时率 |

## 三、部门归属

指标 owner 归到部门层：

| Department | 主要负责的域 |
|---|---|
| **风险管理部** (D_RISK) | risk + credit 部分 |
| **财务部** (D_FIN) | finance |
| **自营信贷产品部** (D_SELF) | credit（SELF_LOAN） |
| **平台业务产品部** (D_PLAT) | credit（PLATFORM_LOAN） |
| **联合贷分润业务部** (D_JOINT) | credit（JOINT_LOAN）+ funding |
| **营销部** (D_MKT) | marketing |
| **运营部** (D_OPS) | user + operation |
| **数据部** (D_DATA) | 全部指标 Steward |
| **法务合规部** (D_LEGAL) | compliance |
| **客服中心** (D_CS) | operation（客服） |

## 四、Level × Domain 交叉矩阵（示例）

|  | user | credit | finance | risk | marketing | operation |
|---|---|---|---|---|---|---|
| **L1** | DAU、MAU、新客数 | 放款额、通过率、支用率 | 净利润、总收入 | M1/M3 逾期率、拨备覆盖 | ROI、总获客成本 | 员工人均产能 |
| **L2** | 留存率、7 日留存 | 各产品放款、户均金额 | 净息差、各产品毛利 | Vintage、反欺诈拦截 | 各渠道 CPA、转化率 | 客服 CSAT |
| **L3** | 各年龄段活跃 | 各分公司业绩 | 单位客户成本 | 各风险等级客户数 | 单个活动 ROI | 分公司业绩 |

## 五、监管指标标记

监管报送用的指标，需在 `metrics.yaml` 中标记：

```yaml
- id: M0001
  is_regulatory: true
  regulator: 银保监会
  filing_frequency: monthly
```

变更需 CRO 额外审批。

## 六、维度体系

指标可下钻/切分的维度：

- **时间维**：小时、天、周、月、季度、年
- **产品维**：4 大产品 + 具体子产品
- **渠道维**：10 个 channel_code
- **组织维**：全公司 → 部门 → 团队 → 员工
- **地域维**：全国 → 分公司 → 城市 → 省份
- **客户维**：年龄段、性别、职业、学历、风险等级、客户等级

## 七、指标 vs 报表 vs 看板

三者关系：
- **指标**（Metric）：原子数据点，一个业务度量
- **报表**（Report）：多个指标 + 维度的组合展示
- **看板**（Dashboard）：多个报表 + 图表 + 交互

一个报表典型引用 5–20 个指标；一个看板引用 3–10 个报表。

## 八、指标平台结构

```
指标平台
├── Catalog（目录，metrics.yaml）
├── Registry（注册）
├── Compute Engine（对接 DW）
├── Governance（审批、变更、审计）
├── API（供 BI、Agent、下游调用）
└── UI（业务人员浏览、申请、订阅）
```

看板与报表引用指标平台的 API，永远不会跳过指标平台直接查数仓（除非临时探索）。

## 九、生命周期状态

指标状态字段 `status`：

| 状态 | 说明 |
|---|---|
| `DRAFT` | 草稿，未审批 |
| `ACTIVE` | 上线中，可用 |
| `DEPRECATED` | 已废弃，仅历史可用 |
| `ARCHIVED` | 归档，不可用 |

变化规则：`DRAFT → ACTIVE → DEPRECATED → ARCHIVED`（不可逆）
