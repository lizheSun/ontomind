# 信优消费金融 —— 指标体系管理办法

- **文档版本**：v1.0
- **发布日期**：2025-01-15
- **主责部门**：数据部
- **参与方**：财务部、风险管理部、各业务部门、法务合规部
- **审批人**：CFO + CDO

## 一、总则

### 1.1 目的

本办法规范信优消费金融公司的**指标（Metric）**从**申请、创建、口径变更、废止**的全生命周期治理，确保：
- **单一事实版本（Single Source of Truth）**：同一指标在全公司口径唯一
- **可复用**：一次定义、跨部门共享
- **可追溯**：每次口径变更留痕
- **可验证**：所有指标可通过统一 SQL 模板复现
- **可解释**：为 Data Agent、NL2SQL、指标平台提供标准化元数据（本体基础）

### 1.2 适用范围

- 全公司所有对内、对外报表使用的业务指标
- 数据仓库 DWS/ADS 层产出的所有聚合指标
- BI、驾驶舱、Data Agent 中展示的所有指标
- 监管报送用到的指标

### 1.3 主要原则

1. **唯一原则**：一个业务概念只能有一个指标 ID
2. **命名规范**：中英文名 + 别名，符合《命名规范》
3. **权威口径**：口径由 owner 部门审定，其他部门只能使用不能篡改
4. **公开透明**：所有指标定义在指标平台可查
5. **变更审批**：口径变更必须走审批流

## 二、组织与职责

### 2.1 指标治理委员会 (MGC, Metric Governance Committee)

**成员**：CDO（主席）、CFO、CRO、各业务线负责人、数据部部长
**议事频率**：每月一次
**职责**：
- 审议一级指标的新增/变更/废止
- 仲裁跨部门口径分歧
- 每季度指标质量报告

### 2.2 各方角色

| 角色 | 职责 |
|---|---|
| **指标 Owner** | 定义口径、维护指标定义、响应变更申请 |
| **指标 Steward** | 数据部 DE，实现口径的 SQL、维护 DWS 表 |
| **指标 Consumer** | 使用指标（BI、Agent、下游报表） |
| **Data Governance Team** | 指标平台建设、变更审批工具、审计 |

## 三、指标生命周期

```
[申请] → [评审] → [实现] → [验收] → [发布] → [使用] → [变更/废止]
```

### 3.1 申请

任何员工可提交指标申请，包含：
- 业务名称 + 别名
- 业务定义
- 计算口径（伪代码或 SQL）
- 使用场景
- 责任部门与 Owner
- 计划上线时间

### 3.2 评审

由 Owner 部门 + 数据部 + 相关部门联合评审，通过率不足 60% 则驳回。

一级指标必须过 MGC。

### 3.3 实现

由数据部 Steward 实现：
- 在 DWD/DWS 层加数据
- 编写 SQL 模板
- 单元测试 + 与 owner 提供的参考数据核对
- 在指标平台注册

### 3.4 验收

指标 Owner 抽样 10 个日期验证结果一致后签字，才能发布。

### 3.5 发布

指标进入指标目录（`metrics.yaml`），版本置为 v1.0，状态 `ACTIVE`。

### 3.6 变更

变更类型分三级：

| 级别 | 内容 | 审批 | 影响处理 |
|---|---|---|---|
| **L1 无损** | 别名添加、描述补充 | Owner 单批 | 立即生效 |
| **L2 有损** | 计算口径微调（如加个过滤） | Owner + Steward + Consumer 通知 | 灰度 2 周 + 新旧口径并存 |
| **L3 破坏性** | 口径重大变更、单位变更、语义变更 | MGC 全体 | 建议新建指标；旧指标标记 DEPRECATED，保留 6 个月 |

### 3.7 废止

指标废止后：
- 状态 `DEPRECATED`
- 平台仍可查看历史，但不允许新增引用
- 6 个月后进入 `ARCHIVED`，仅历史用

## 七、审计与合规

- 所有指标定义变更留审计日志（谁、何时、改了什么）
- 每季度指标质量报告：
  - 使用率排行
  - 变更次数
  - 违规使用（未审批就发报表）
  - 数据质量得分
- 监管报送用指标单独标记 `is_regulatory=true`，变更需 CRO 审批

## 八、指标目录（Catalog）

指标定义存于 `metrics/catalog/metrics.yaml`，可通过指标平台 API 查询。

主要字段：
- `id`、`name_zh`、`name_en`、`aliases`
- `level`（一级/二级/三级）、`domain`、`department`、`owner`
- `definition`（业务定义）、`formula`（伪公式）、`filters`（过滤条件）
- `source_tables`、`sql_template`
- `related_metrics`、`version`、`effective_from`、`status`

## 九、与 Data Agent / 本体化语义层的关系

本指标体系是**公司本体化语义层的第一层**（数据字典本体）：
- 每个指标即一个"业务实体"
- 别名 = 同义词表
- SQL 模板 = 从业务概念到物理表的映射
- 关系（`related_metrics`）= 概念间的关联

未来将扩展到：
- 第二层：**指标本体** (metric ontology)
- 第三层：**关系本体**（跨表关联、血缘）
- 第四层：**推理规则本体**（业务规则驱动的自动决策）

见研究文档 `.blueprint/research/ontology-and-data-agent.md`。
